# kubernetis-dz08
Компоненты Kubernetes



Решение разбито на логические шаги: от расчета чистых ресурсов приложения до подбора инфраструктуры с учетом служебного overhead и отказоустойчивости.

## 1. Расчёт необходимых ресурсов для приложений
Для начала определим, сколько ресурсов (CPU и RAM) требуется для работы всех компонентов приложения в нормальном режиме. 

База данных:
  - реплики = 3
  - ram на 1 копию = 4 ГБ
  - cpu на 1 копию = 1 ядро
  - итого ram = 12 ГБ
  - итого cpu = 3 ядра

Кеш:
  - реплики = 3
  - ram на 1 копию = 4 ГБ
  - cpu на 1 копию = 1 ядро
  - итого ram = 12 ГБ
  - итого cpu = 3 ядра

Frontend:
  - реплики = 5
  - ram на 1 копию = 0.05 ГБ (50 МБ)
  - cpu на 1 копию = 0.2 ядра
  - итого ram = 0.25 ГБ
  - итого cpu = 1 ядро

Backend:
  - реплики = 10
  - ram на 1 копию = 0.6 ГБ (600 МБ)
  - cpu на 1 копию = 1 ядро
  - итого ram = 6 ГБ
  - итого cpu = 10 ядер

**Итого по всем приложениям (21 под):**
  - итого ram = 30.25 ГБ
  - итого cpu = 17 ядер

  
## 2. Выбор размера рабочей ноды и расчёт количества (Worker Nodes)
Согласно [best practices по выбору размера нод](https://learnk8s.io/kubernetes-node-size), "золотой серединой" являются ноды с 4–8 vCPU. Слишком маленькие ноды увеличивают расходы на служебные поды, а слишком большие создают огромный "blast radius" (зону поражения) при падении. 

Выберем стандартный размер облачной ноды: 8 vCPU / 32 ГБ RAM.


### Учёт отказоустойчивости (Reserve)
По условию, кластер должен пережить выход из строя одной ноды. Это значит, что оставшиеся в живых ноды должны суммарно иметь достаточно ресурсов для размещения всех 21 подов.

Формула для расчёта: 
`Allocatable ресурсы одной ноды * (N - 1) >= Суммарные ресурсы приложений`

Где `N` — общее количество нод.
* **По CPU:** `8 * (N - 1) >= 17`  =>  `N - 1 >= 2.12`  =>  `N - 1 = 3`  =>  **N = 4**.
* **По RAM:** `32 * (N - 1) >= 30.25`  =>  `N - 1 >= 0.94`  =>  `N - 1 = 1`  =>  **N = 2**.

Чтобы удовлетворить оба условия, берём максимальное значение. Нам нужно **3 выжившие ноды**, следовательно, общее количество рабочих нод **N = 4**.

## 3. Учёт служебных ресурсов (System Overhead)
Операционная система, процессы kubelet, container runtime и пороги высвобождения памяти (eviction thresholds) потребляют ресурсы ноды. K8s не может отдать 100% "железа" подам.

Согласно [документации K8s](https://kubernetes.io/docs/tasks/administer-cluster/reserve-compute-resources/), необходимо зарезервировать ресурсы:
* `system-reserved` (для ОС): ~500m CPU, 1 ГБ RAM.
* `kube-reserved` (для kubelet, containerd): ~500m CPU, 1 ГБ RAM.
* `eviction-threshold`: 100m CPU, 0 ГБ RAM (но зарезервируем чуть больше для стабильности).


**Зарезервируем на каждой Worker ноде:** `1 vCPU` и `2 ГБ RAM`.


### Пересчёт с учётом служебных ресурсов (Allocatable)
* **Доступно для подов на 1 ноде:** 7 vCPU и 30 ГБ RAM.
* **Проверка условия отказа 1 ноды:**
  Если 1 нода падает, остаются 3 ноды. Их суммарный *allocatable* ресурс:
  * CPU: `3 * 7 = 21 vCPU` (Нам нужно 17. **Условие выполняется: 21 > 17**)
  * RAM: `3 * 30 = 90 ГБ` (Нам нужно 30.25. **Условие выполняется: 90 > 30.25**)

**Итого Worker Nodes:** 4 ноды конфигурации **8 vCPU / 32 ГБ RAM**.

## 4. Расчёт ресурсов для Control Plane (Master Nodes)

Для Control Plane (master nodes) не нужны огромные объёмы RAM, но критически важна отказоустойчивость (для etcd требуется нечётное количество нод, минимум 3) и быстрые CPU для API-сервера.

Для кластера такого размера (до 10 нод, ~50 подов с учетом системных) достаточно следующих параметров:
* **Количество:** 3 ноды (для HA etcd и отказоустойчивости API).
* **Параметры:** 2 vCPU / 4 ГБ RAM на ноду.
* **Служебные ресурсы (etcd, API server, scheduler):** Зарезервируем `1 vCPU` и `2 ГБ RAM` на ноду.
* **Доступно для системных подов (CoreDNS, Ingress, CNI, Monitoring):** 3 vCPU и 6 ГБ RAM (суммарно на 3 ноды), чего более чем достаточно.

**Итого Master Nodes:** 3 ноды конфигурации **2 vCPU / 4 ГБ RAM**.

| Тип ноды | Количество | Параметры (CPU/RAM) | Назначение |
| :--- | :---: | :--- | :--- |

### Рекомендации по упаковке в Helm Chart
Так как по условию приложение необходимо упаковать в Helm Chart для деплоя в разные окружения, рассчитанные ресурсы должны быть зашиты в файлы `values.yaml` и шаблоны `deployment.yaml`  `statefulset.yaml`:

1. **Requests и Limits:** Для всех компонентов в чарте необходимо прописать `resources.requests` и `resources.limits`, равные рассчитанным значениям (например, для Backend: `cpu: 1`, `memory: 600Mi`). Это гарантирует, что `kube-scheduler` сможет корректно распределить поды по нодам и не допустит OOMKill.
2. **Гибкость (Values):** Количество реплик (`replicaCount`) и размеры ресурсов нужно вынести в `values.yaml`, чтобы иметь возможность масштабировать их под разные окружения (Dev, Stage, Prod) без изменения самих шаблонов.
3. **Anti-Affinity:** Для отказоустойчивых компонентов (БД и Кеш, у которых по 3 копии) в чарте необходимо добавить правила `podAntiAffinity`. Это заставит `kube-scheduler` рассадить 3 реплики БД и 3 реплики Кеша по **разным** рабочим нодам. Если одна нода упадет, мы потеряем только 1 реплику, а сервис продолжит работать.

p.s. Расчёт выполнен с учётом материалов [Considerations for large clusters](https://kubernetes.io/docs/setup/best-practices/cluster-large/) и [Architecting Kubernetes clusters — choosing a worker node size](https://learnk8s.io/kubernetes-node-size).
